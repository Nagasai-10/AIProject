# Week 4 Pipeline Task Dependency Map

## Context
Week 4 is the point where the data pipeline becomes operational. To avoid confusion, the work needs an explicit dependency order.

## Week 4 objective
The objective of this note is to turn the data pipeline into a visible sequence of work packages that GitHub can track.

## Week 4 dependency order

1. dataset access confirmed  
2. class and folder audit completed  
3. file-quality observations logged  
4. canonical structure fixed  
5. manifest expectations defined  
6. split-control rules documented  
7. preprocessing assumptions recorded  
8. downstream handoff to architecture, training, evaluation, and ethics streams

## Why this matters
Without a dependency map, data work becomes scattered across notebooks and personal assumptions. This note ensures Week 4 decisions can be translated into issues, commits, and later scripts.

## Week 5 handoff direction
In Week 5, these dependency steps should become final split scripts, manifests, and pipeline-ready directory outputs.

## Summary
This Week 4 note makes the data stream operational by turning it into an ordered, trackable workflow.
